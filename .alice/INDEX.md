## 🧠 AI 語意判斷原則

**不再使用關鍵詞匹配路由表。** AI 應基於任務的語意本質自行判斷需要哪些資訊：

1. **這個任務會觸及哪些系統邊界？** → 讀 `FACTS.md`（架構邊界 / 設計決策段落）
2. **這個任務是新增功能還是修改現有？** → 讀 `TASK_BOARD.md`（確認目前進度、避免重工）
3. **近期是否有相關變更需要了解？** → 讀 `LOG.md`（預設讀末 10 條，必要時回溯）
4. **是否需要理解模組間的依賴關係？** → 讀 `ARCHITECTURE.md`（資料流、關鍵檔案）
5. **任務涉及多個專案領域？** → 讀 `FACTS.md` 全文 + `TASK_BOARD.md` 相關段落 + `LOG.md` 末 10 條

判斷完後，只讀取真正需要的文件段落，不浪費 token。

## 使用協議

0. **核心心法**：「讀資訊、看進度、懂需求」——每次修改前，先理解架構邊界（FACTS.md）、掌握開發進度（TASK_BOARD.md）、了解近期變更脈絡（LOG.md）。這不是選項，是所有行動前的強制思維流程。

1. **觸發時機**：任何 `overwrite_file`、`replace_in_file`、`apply_diff`、`create_or_update_skill` 或架構決策前

2. **讀取順序**：INDEX.md → AI 語意判斷需要哪些文件 → 讀取判斷所需的文件 → 確認無衝突 → 執行

3. **回報格式**：執行前在思考中確認「已讀 `.alice/` → 已讀資訊、看進度、懂需求 → 無邊界衝突」

4. **ADSP 異動報告 = LOG.md 條目**：每次執行 ADSP 協議的 `overwrite_file` / `replace_in_file` / `apply_diff` 後，提交給主人的異動報告摘要**必須同步寫入 `LOG.md`**。異動報告與日誌為原子操作——不可只寫報告不寫日誌，也不可只寫日誌不寫報告。

5. **新建前強制建立進度條目**：任何新 Skill、新功能、新工具建立前，必須先在 `TASK_BOARD.md` 對應專案區塊新增 `🔧 進行中` 條目（含需求描述、邊界檢查要點、預期影響範圍）。完成開發後更新為 `✅ 完成` 並補寫 `LOG.md`。

6. **自檢一致性**：LiveCode Phase 0 自檢時，應交叉驗證 `TASK_BOARD.md` 中 `✅ 完成` 的項目是否在 `LOG.md` 中有對應的變更紀錄。若不一致，標記為異常。
